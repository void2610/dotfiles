# コメント分類タクソノミー

Phase 2 で収集した各レビューコメントに対して、以下 3 軸で分類する。

## 1. 対応方針 (必須)

| 記号 | 意味 | 扱い |
|------|------|------|
| 🔴 | 必須修正項目 (blocking) | Phase 6 で必ず修正 |
| 🟡 | 推奨修正項目 (suggestion) | Phase 6 で修正 |
| 🟢 | 質問・議論項目 (question/discussion) | 修正対象外。必要なら返信のみ |

Phase 3 の修正計画では 🔴 と 🟡 のみを対象とする。

## 2. 技術的重要度 (参考)

| 記号 | 意味 |
|------|------|
| 🚨 | クリティカル (システム障害リスク) |
| 🔥 | 高 (本番環境への影響大) |
| 📈 | 中 (品質・保守性への影響) |
| 📝 | 低 (スタイル・可読性) |
| 🎯 | 改善提案 (将来的な拡張性) |

## 3. カテゴリー

| カテゴリー | 典型例 |
|-----------|-------|
| 仕様適合性 | 要件定義との整合性、既存機能の仕様、受け入れ条件 |
| コーディングルール | `docs/` との整合性、命名規則、インデント・フォーマット |
| パフォーマンス | メモリ最適化、クエリ効率、API 呼び出し最適化、キャッシュ戦略 |
| アーキテクチャ | 類似実装との統一性、ADR との整合性、依存方向、デザインパターン |
| テスト品質 | カバレッジ、ケース網羅性、モック適切性、可読性・保守性 |
| エラーハンドリング | 例外処理、エラーメッセージ、ログ出力、復旧可能性 |
| 可読性・保守性 | 自己文書化、コメント、責務分離、複雑度管理 |
